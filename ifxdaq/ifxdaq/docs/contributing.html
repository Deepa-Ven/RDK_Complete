How To Contribute

First off, thank you for considering contributing to ifxdaq!

This document intends to make contribution more accessible by codifying tribal knowledge and expectations. Don’t be afraid to open half-finished PRs, and ask questions if something is unclear!

Local Development Environment

Python

We use Python to develop our package. The main development version is Python3.11 (supported versions: 3.8-3.11). Please install this version according to your OS.

Hint

pyenv (installable with scoop under Windows) is an excellent tool to manage different Python versions on your machine.

Clone repository

First, you need a copy of the code. The code base is hosted in a Gitlab repository. We use git to version control our code. You can clone the repository with:

git clone git@gitlab.intra.infineon.com:pss/sensys/algo/ifxdaq.git

Install & configure toolchain

Afterwards run the setup.sh script inside the repository, which will automatically:

  1. Configure git.

  2. Check that Python3.11 is available.

  3. Create a local virtual environment.

  4. Install all required Python packages.

  5. Install pre-commit hooks.

On Windows run:

git bash -c "bash setup.sh; read -p 'Press [Enter] to close this window...'"

On Unix run:

bash setup.sh

Every time you start a new console and want to use the virtual environment, you have to activate it.

Style guide

A good orientation for a style guide is the Google Style Guide. If you stick roughly to those guidelines, your code style is probably already quite nice!

Here we want to point out a few important topics for this project that are already contained in the same manner or very similar in Google’s style guide.

Code style

We use black and ruff to format our code automatically, so there is no need to pay special attention to this topic. (Note: Integrating those tools into your IDE might be really helpful.)

Typing

Type annotations improve the readability and maintainability of your code. The type checker will convert many runtime errors to build-time errors. This helps to avoid obvious errors and to create cleaner APIs for our functions. You can find a nice cheat sheet here.

Documentation

Contributed code should be documented already during creation. This simplifies cooperation between multiple developers and also re-use in other projects. We follow the Google Docstring style to document our code inline. Here you find an extensive example of this style.

You can also add explicit documentation under docs. Just have a look into this folder to find examples how this is done.

Testing

Add tests for your code. This will ensure high quality and simplifies the development, when the complexity of the project increases. Tests are added under tests. Have a look into already existing tests to get an impression, how this can be done.

git

Effective communication makes software scale. Git is our medium and a very effective one to communicate our changes. Your changes should tell a story that’s easy to follow, so that the code can easily be picked up by others by a mere examination of its commit and list history. There needs to be structure to the changes that are sent, so that they can be easily reviewed, and held to high standards.

Commit message conventions

Commit message format Each commit message consists of a header, a body, and a footer.

<header>
<BLANK LINE>
<body>

Commit message header format

<type>: <short summary>
   |         |
   |         └─ Imperative, present tense. Not capitalized.
   |            No period at the end. Less than 50 characters.
   |
   └─ Commit Type: ci|docs|feat|fix|refactor|test

  • ci: Changes to our CI configuration files and scripts (example scopes: Gitlab)

  • docs: Documentation only changes

  • feat: A new feature

  • fix: A bug fix

  • refactor: A code change that neither fixes a bug nor adds a feature

  • test: Adding missing tests or correcting existing tests

Commit message body format

  • Use the imperative, present tense (e.g. ‘fix’ not ‘fixed’, ‘fixes’).

  • Wrap at 72 characters.

  • Use the body to explain what and why vs. how. Explain the motivation for the change.

  • You can include a comparison of the previous behavior with the new behavior in order to illustrate the impact of the change.

  • You can reference issues with #SENSYS-<issue number>.

Commit message template

.gitmessage will pre-populate your commit message with an empty body, which shows you how a commit message should look like. The .gitmessage template will be automatically installed with the setup.sh script.

Branching conventions

We use the git-flow branching model. Please find a detailed description of the branching model in Bitbucket Tutorial and Vincent Driessen’s Blogpost.

Branch name format

<type>/SENSYS-<issue-number>-<short description>
   |                |               |
   |                |               |
   |                |               └─ Short, crisp description. Hyphen as separator between words.
   |                |
   |                └─ Issue number referencing the issue.
   |
   └─ Commit Type: bugfix|feature|hotfix|release

Tools

All used tools are open-source and have a pretty good online documentation. So, if you get stuck at some point, there is a high chance that somebody else had already the same problem and you find a solution in the documentation or on stackoverflow :)

All tools can be executed locally & are also executed automatically on every push by our CI infrastructure. We created shortcuts to use the tools with invoke and to automate our development infrastructure. invoke is a Python task execution tool & library. invoke provides shortcuts with default parameters that are adequate for most situations, so you don’t have to think a lot about command line arguments. You can list all available shortcuts with:

# List available options:
inv -l

# Run a specific command, e.g. build documentation
inv docs

The listed options will be explained in the following sections.

pre-commit

We use pre-commit to check commits before they are published. The available hooks will check your files for correct formatting, syntax errors and more. pre-commit and its hooks are automatically installed, during the setup of the development environment.

Documentation - sphinx

The documentation generation is built on sphinx. If you want to build the documentation locally, run:

inv docs

The built documentation can then be found in build/docs. The documentation of released version can be accessed at Artifactory (Version > 3.2.4).

Formatting - black

Our code is automatically formatted with black. There is no need to run the formatters manually, since they will be executed by our pre-commit hooks. If you want to do it nevertheless, here are the commands.

# Run black (-c to check the files only)
inv black

Static code analysis - mypy, pylint, ruff

We perform static code analysis to detect simple errors, force high quality code and ensure some common code style. You can invoke the complete static analysis with:

inv static

In the following subsections, we will explain the tools used in the static code analysis and show how to invoke them separately.

mypy

You can use mypy to perform a static type checking of your code. This supports you to find incompatible function calls and function arguments without the need to actually run the code:

inv mypy

pylint

pylint is used to check your code for programming errors, helps to enforce a coding standard, sniffs for code smells and offers simple refactoring suggestions. It is invoked by:

inv pylint

ruff

ruff is an extremely fast Python linter to check your Python code. You can perform the check with:

inv ruff

Testing - pytest & coverage

We use pytest & coverage for running the test suite. pytest is used for the actual testing, while we use coverage to generate reports for our test coverage. Here is a short overview of important commands to run tests (find more information and more examples in the pytest documentation):

# Run all tests
pytest tests
# Run only a specific test
pytest tests -k name_of_the_test
# Run only a subset of tests, located in a specific folder
pytest tests/test_preprocessing

If you simply want to run all the tests & create a coverage report run:

# All tests
inv test -r
# Only unittests
inv unittest -r
# Only system tests
inv systest -r

Test and coverage reports will be available under build/test, build/unit and build/system.

Release

In this section, we describe the steps that have to be performed to create a release:

  1. Verify that all firmware versions in the documentation are up-to-date.

  2. Create a release branch. Naming convention: release/<VERSION>, e.g release/1.2.3.

  3. Update the changelog (CHANGELOG.md) and the versions of the batches in README.md & docs/index.rst.

  4. Create a merge request from the release branch to master.

  5. After approvals, merge the pull request with a Merge commit.

  6. Verify that all build jobs on master pass (and optionally, but recommended, verify the package locally with connected sensors - a quick manual evaluation & running the tests locally is enough).

  7. Tag the master branch with the version number, e.g. git tag v1.2.3 -m "Release 1.2.3", and push the tag, e.g git push origin v1.2.3. This triggers automatically the release & the result will be published on Artifactory.

  8. Checkout the develop branch and rebase it onto master, git pull origin master --rebase.
